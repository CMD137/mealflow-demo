# MealFlow 优惠券修正后探索式复测报告

测试日期：2026-07-13  
测试范围：后台优惠券新增、单一秒杀券类型收敛、非法参数、用户端领券、Redis 秒杀库存边界、构建校验。  
测试方式：不按固定主链路脚本执行，围绕上轮漏测缺陷进行探索式验证。

## 1. 复测结论

后台新增优惠券 `AMOUNT_OFF` 导致 500 的问题已修复。当前后台新增券表单不再暴露可编辑类型文本框，类型固定为“秒杀券”；接口非法类型也能返回 400 业务错误。

同时发现新的 P1 缺陷：Redis 秒杀模式下防超卖生效，但后台和用户端展示的库存仍读取数据库旧值，领取后不会随 Redis 剩余库存变化。

## 3. 新发现问题

| 优先级 | 问题 | 现象 | 原因分析 | 建议 |
| --- | --- | --- | --- | --- |
| P1 | Redis 秒杀库存与页面/后台库存展示不一致 | 券 `10003` 初始库存 2；两个用户领取成功，第三个用户返回 `SOLD_OUT`；但后台 `/vouchers/admin` 仍返回 `stock = 2`，用户端券列表也仍显示 `库存 2` | Docker 环境配置 `PROMOTION_SECKILL_MODE=redis`。Redis Lua 扣减的是 `voucher:stock:{voucherId}`，数据库 `voucher.stock` 没有同步扣减；列表接口仍读取数据库库存 | 后台和用户端展示应使用 Redis 剩余库存，或在领取成功后同步数据库库存；否则运营和用户看到的库存会误导 |
| P3 | 用户端仍保留“普通券”兜底文案 | `meal-user-web/src/views/VouchersView.vue` 中仍有 `voucher.type === 'SECKILL' ? '秒杀券' : '普通券'` | 产品已决定未来只保留秒杀券，普通券概念应从前端文案中移除 | 改为固定显示“秒杀券”，避免未来出现脏数据时重新暴露普通券概念 |

## 4. 复测数据

## 5. 结论

本轮修复解决了“后台新增优惠券默认类型错误导致 500”的问题，也落实了只保留 `SECKILL` 秒杀券的方向。当前阻塞点转移为库存展示一致性：Redis 秒杀扣减和数据库库存展示不同步，会让后台运营和用户端看到错误库存，应作为下一轮优先修复项。
